# Delivery Checklist: Profile + Opportunity Pack

Use this checklist to deliver a fixed-scope Profile + Opportunity Pack without turning it into an open-ended consulting engagement.

## Before Research

- Confirm the client's public GitHub profile URL.
- Confirm the one public repository README to review.
- Confirm preferred languages, tools, and task types.
- Confirm weekly contribution time.
- Confirm work boundaries and topics to avoid.
- Confirm the client understands this pack does not guarantee income, merged PRs, bounty awards, sponsorship, stars, or leads.

Do not request passwords, private tokens, payment account access, production credentials, or private repository access for this package.

## Research

- Review the first screen of the GitHub profile.
- Review pinned or prominent repositories if visible.
- Review the target repository README for install/run clarity, proof of output, audience fit, and call to action.
- Run or review an opportunity scan matched to the client's skills.
- Manually inspect top issue candidates before recommending action.
- Reject unsafe, already-claimed, noisy, stale, vague, or platform-risk matches.

## Deliverables

The final pack should include:

- client snapshot
- positioning recommendation
- profile README suggestions
- target repository README suggestions
- short intro or service-offer draft
- 10-15 screened opportunities when enough good matches exist
- top 3 recommended actions
- rejected-match notes
- first-week action plan
- follow-up options

Use `templates/profile_opportunity_pack_template.md` as the delivery format.

## Quality Bar

Do not deliver the pack if:

- the profile recommendation is generic and not tied to the client's public proof
- the repository README advice does not include concrete changes
- all recommended opportunities are stale, unsafe, assigned, or unclear
- the report implies guaranteed income or guaranteed maintainer acceptance
- the client would need to spam maintainers to act on the recommendations

## After Delivery

- Offer one focused follow-up, such as README rewrite, PR preparation, or a small automation script.
- Do not comment on issues, open pull requests, claim bounties, or make account-bound commitments without explicit client approval.
- Keep any follow-up scope small, written, and acceptance-based.
